# Work Report → Revenue 실적 자동집계 관련 의문점/주의사항

## A. 직책 ↔ 등급 매핑 관련

1. **직책 변경 이력 처리** — 사원이 프로젝트 진행 중 승진(예: 과장 → 차장)하면 그 시점 이전·이후 Work Report는 어느 등급으로 집계해야 하나? (작성 당시 직책? 현재 직책?)
2. **퇴사자 Work Report** — 퇴사한 사원의 과거 Work Report도 등급 조회가 가능해야 한다. 직원마스터에서 퇴사자도 직책 이력이 보존되는지 확인 필요.
3. **매핑표 외 직책** — 책임/수석/선임/주임연구원 같은 R&D 계열 직책이나 사외이사·감사 등은 어느 등급에 매핑되나? 매핑되지 않으면 집계 누락 위험.
4. **직책 미설정 사원** — 신입 입사 직후나 직책이 비어 있는 사원의 Work Report는 누락되나, 별도 "미분류" 등급으로 집계되나?

## B. Work Report 데이터 필터링

5. **업무구분=프로젝트가 아닌 행** — 부서업무·연차·기타로 기록된 시간은 제외해야 함. 단, "제안" 구분(WRD0001 등)은 Pre-Sales 단계 활동이므로 어떻게 처리할지 명확화 필요.
6. **프로젝트명 미선택 행** — 업무구분이 프로젝트인데 프로젝트명을 선택하지 않은 행이 있으면 어디로도 집계되지 않음. 이런 데이터의 처리 정책 필요.
7. **삭제된/미작성 Work Report** — Work Report 상태가 "미작성"인 날의 공수는 0으로 집계되는가, 아니면 집계 대상에서 제외되는가?
8. **수정 시 재계산** — 과거 Work Report를 수정하면 해당 월 실적이 즉시 재계산되어야 한다. 캐싱 무효화 정책 필요.

## C. M/D 환산 정확성

9. **480분 = 1 M/D 가정의 타당성** — 회사의 표준 근무시간이 8시간(480분)이 맞는지, 4시간(반차)·기타 변동 근무 처리 어떻게 되는지 확인.
10. **합산 단위 정밀도** — 0.25, 0.125 단위 소수점 M/D를 어디까지 표시할 것인가? (반올림 자리수)
11. **하루 480분 초과 입력 가능성** — 한 사원의 하루 소요시간 합이 480분을 넘어가면 1 M/D 초과 발생. 정책상 허용되는지, 캡(cap)을 씌워야 하는지.

## D. 등급별 단가 일관성

12. **단가 변경 시점** — 등급별 단가(특급 1천만/고급 900만/중급 700만/초급 500만)가 회계연도 중 바뀌면 과거 실적 금액은 당시 단가로 잠금되는지, 신단가로 재계산되는지?
13. **프로젝트별 단가 차이** — 일부 프로젝트는 별도 단가 협약이 있을 수 있다. 프로젝트별 단가 오버라이드를 지원해야 하나?
14. **외주 vs 내부인력 단가 분리** — 외주 인력의 등급별 단가는 별도일 텐데, Work Report에서는 내부인력만 집계 대상으로 명확히 한정해야 함.

## E. 모달 UI/UX 측면

15. **자동집계 vs 수동입력 충돌** — 사용자가 실적 행의 월별 금액을 수동 수정한 뒤 Work Report가 새로 들어오면 덮어쓰는가, 보존하는가? "수동 잠금" 마커 필요할 수 있음.
16. **등급 행이 없을 때** — Pre-Sales 계획에 특급 행이 없었지만 실적은 특급이 발생한 경우, 실적 모달에 특급 행이 자동 생성되어야 하나? extra 행으로 추가?
17. **계획 0 / 실적 발생** — 차이 섹션에서 "계획=0, 실적>0"인 경우 차이가 음수로 잘못 표시될 위험. 표기 정책 필요.
18. **실적 행의 수량/공수 셀** — 현재 모달에서 실적 행의 수량 셀은 disabled. 자동집계 도입 시 표시는 하되 수정 불가로 유지할지, 합산 결과를 수량 셀에 채워 보여줄지 결정 필요.

## F. 권한/보안

19. **다른 사원 Work Report 조회 권한** — 실적 합산을 위해 시스템이 모든 사원의 Work Report를 읽어야 한다. 일반 사용자가 모달을 열 때도 합산값이 보이는데, 이게 개인 정보 노출이 되지 않는지 검토.
20. **PM 외 권한자가 Work Report를 수정** — 프로젝트 PM이 아닌 사람이 다른 프로젝트 Work Report를 수정해도 실적에 반영되는지, 승인 절차 필요한지.

## G. 시점/성능

21. **언제 합산을 수행하나** — 모달 열 때마다 실시간 집계? 야간 배치? 트리거 기반?
22. **장기 프로젝트 성능** — 1년 이상 프로젝트의 Work Report 행 수가 수만 건이 될 수 있다. 모달 응답 시간 보장 방안.
23. **월 마감 후 데이터 수정** — 월 마감(closing) 처리 후 과거 월의 Work Report가 수정되면 재무 보고서 영향. 마감 잠금 정책 필요.

## H. 검증 포인트

24. **공수비교표와의 일치** — Project Management View 공수비교표의 "Work Report" 컬럼 합계와 Revenue 모달 실적 합산의 M/D 합계가 반드시 일치해야 함. 불일치 시 어디가 정답인지.
25. **VRB 예상 공수와 실적의 비교** — Pre-Sales 계획 M/D vs Work Report 실적 M/D 차이가 큰 경우 알림/경고 필요한지.
26. **이중 집계 방지** — 동일 사원이 동일 일자에 동일 프로젝트로 여러 행을 작성한 경우 모두 합산되는데, 이게 의도인지(여러 task 분할 입력) 중복인지 검증 어렵다.

## I. 데이터 모델

27. **사원 → 부서 변경** — 사원이 프로젝트 도중 부서 이동을 하면 직책·등급은 유지되지만 부서 기준 통계가 흔들린다. 영향 범위 확인.
28. **사외/파견 인력의 직책** — 자사 사원이 아닌 파견·계약 인력이 Work Report를 작성하는 경우 등급 매핑 가능 여부.
